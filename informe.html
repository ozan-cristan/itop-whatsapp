1. Resumen ejecutivo

El iTop WhatsApp Bot es un sistema de atención al cliente que integra el canal de mensajería WhatsApp con la plataforma ITSM iTop de Angel Baraldo. Permite que los usuarios finales (personal de la empresa o clientes con contrato de servicio) gestionen sus solicitudes de soporte técnico sin necesidad de acceder al portal web de iTop.

El bot se identifica por el número +54 341 781-3171 y se conecta a la infraestructura oficial de Meta WhatsApp Business Platform API v22.0. Toda comunicación pasa por servidores de Meta antes de llegar al webhook del bot, garantizando que el cifrado extremo a extremo de WhatsApp se mantenga intacto.

Funcion

Creacion de tickets

Guia al usuario paso a paso para crear un UserRequest en iTop, eligiendo familia de servicio, servicio, subcategoria, titulo y descripcion. Soporta adjuntos de archivos.

Funcion

Consulta de solicitudes

Lista todos los tickets activos o resueltos del usuario, con estado en tiempo real y detalle de la ultima nota publica del equipo tecnico.

Funcion

Comentarios

Permite al usuario agregar comentarios publicos a un ticket existente directamente desde WhatsApp, sin necesidad de portal web.

Seguridad

Identificacion por CUIT

El usuario se autentica ingresando su CUIT, que es validado contra el campo personalizado de iTop. La sesion expira automaticamente a los 15 minutos de inactividad.

Repositorio: https://github.com/ozan-cristan/itop-whatsapp — rama feature/meta-api

2. Arquitectura del sistema

El sistema sigue una arquitectura de tres capas: la capa de mensajeria (Meta), la capa de aplicacion (Node.js/Express), y la capa de datos (iTop ITSM). Apache actua como proxy reverso HTTPS entre internet y la aplicacion interna.

Flujo de un mensaje entrante

  1. El usuario envia un mensaje de texto, imagen o documento por WhatsApp.
  2. Meta entrega el payload cifrado mediante una solicitud HTTP POST al webhook configurado: https://postventa.baraldoargentina.com.ar/webhook.
  3. Apache (con SSL) recibe la solicitud y la reenvía via mod_proxy al puerto interno 3000.
  4. Express responde inmediatamente con HTTP 200 para evitar reintentos de Meta, y procesa el mensaje de forma asincrona.
  5. Se adquiere un mutex por usuario (numero de telefono) para serializar mensajes concurrentes.
  6. La maquina de estados (flow.js) determina la respuesta segun el estado actual de la sesion.
  7. Si se requiere interaccion con iTop, el modulo itop.js ejecuta la llamada REST correspondiente.
  8. El bot envia la respuesta al usuario a traves de la API de Meta (graph.facebook.com/v22.0/<PHONE_NUMBER_ID>/messages).

3. Componentes del proyecto

bot.js

Servidor Express + Webhook

Punto de entrada principal. Gestiona la verificacion y recepcion del webhook de Meta, serializa mensajes por usuario con async-mutex, descarga adjuntos multimedia y envia respuestas. Usa la API de Meta Graph v22.0 para envios.

flow.js

Maquina de estados

Implementa la logica conversacional completa con 15 estados. Recibe el mensaje del usuario y el estado actual de la sesion, y retorna el mensaje de respuesta correspondiente. Construye los payloads de botones interactivos de Meta.

state.js

Gestion de sesiones

Almacena el estado de cada conversacion en un Map en memoria con TTL de 15 minutos. Expone funciones CRUD para sesiones: createPendingSession, updateSession, endSession, clearSession.

itop.js

Cliente iTop REST API

Encapsula todas las llamadas a la REST API de iTop v1.3. Autenticacion mediante Application Token. Incluye 10 funciones de dominio que ejecutan operaciones OQL contra las clases de iTop: Person, Service, UserRequest, Attachment.

4. Flujo de conversacion

El bot implementa una maquina de estados finitos con 15 estados. La transicion entre estados esta determinada por el input del usuario y el contexto de la sesion. El comando cancelar puede interrumpir cualquier flujo (excepto la identificacion inicial) y retornar al menu principal.

7. Operaciones iTop REST API

Todas las operaciones se realizan contra /webservices/rest.php?version=1.3 usando el metodo HTTP POST con content-type application/x-www-form-urlencoded. El campo json_data contiene el payload en formato JSON.

FuncionOperacion iTopClase iTopDescripcion
findPersonByCuil() core/get Person Busca un usuario por el campo personalizado cuit. Retorna { id, friendlyname, org_id, org_name } o null.
findPersonByMobile() core/get Person Busca por mobile_phone probando con y sin prefijo +. Funcion alternativa no usada en el flujo principal actual.
getFamiliesForOrg() core/get ServiceFamily Retorna familias de servicios con al menos un servicio activo en los contratos de cliente (CustomerContract) de la organizacion.
getServicesForOrgAndFamily() core/get Service Lista servicios de una familia especifica vinculados a la organizacion via contratos activos. Excluye servicios obsoletos.
getServicesForOrg() core/get Service Lista todos los servicios de la organizacion sin filtrar por familia. Funcion auxiliar disponible.
getSubcategoriesForService() core/get ServiceSubcategory Retorna las subcategorias activas de un servicio dado. Excluye subcategorias con estado obsolete.
getTemplateForSubcategory() core/get RequestTemplate Verifica si existe una plantilla para el par servicio + subcategoria. Si existe, el flujo informa al usuario que debe usar el portal web.
createUserRequest() core/create UserRequest Crea el ticket con campos: org_id, caller_id, service_id, servicesubcategory_id, title, description, origin: 'chat'. Retorna { id, ref }.
attachToTicket() core/create Attachment Sube un adjunto al ticket. El archivo llega en base64 desde la descarga previa de la API de Meta. Campos: item_class, item_id, contents { data, filename, mimetype }.
getTicketsForPerson() core/get UserRequest Lista tickets activos (excluyendo closed y resolved) del usuario. OQL con JOIN a Person.
getResolvedTicketsForPerson() core/get UserRequest Lista tickets con estado resolved del usuario.
getTicketDetail() core/get UserRequest Obtiene detalle completo: ref, titulo, estado, nombre del equipo (team_id_friendlyname), tecnico (agent_id_friendlyname) y ultima entrada del public_log.
addCommentToTicket() core/update UserRequest Agrega una nota publica al ticket usando el mecanismo add_item del campo public_log. El comentario queda visible para el tecnico en iTop.

9. Configuracion Apache

Apache actua como proxy reverso HTTPS, terminando el SSL y reenviando las solicitudes al puerto interno 3000 donde escucha Node.js. Solo se proxea la ruta /webhook.

Configuracion del VirtualHost HTTPS

# /etc/apache2/sites-available/postventa.baraldoargentina.com.ar-le-ssl.conf

<VirtualHost *:443>
  ServerName  postventa.baraldoargentina.com.ar
  ServerAdmin webmaster@baraldoargentina.com.ar
  DocumentRoot /var/www/html

  # ─── SSL ───────────────────────────────────────────
  SSLEngine on
  SSLCertificateFile    /etc/letsencrypt/live/postventa.baraldoargentina.com.ar/fullchain.pem
  SSLCertificateKeyFile /etc/letsencrypt/live/postventa.baraldoargentina.com.ar/privkey.pem
  Include /etc/letsencrypt/options-ssl-apache.conf

  # ─── Proxy al bot (solo /webhook) ──────────────────
  ProxyPreserveHost On
  ProxyPass        /webhook http://localhost:3000/webhook
  ProxyPassReverse /webhook http://localhost:3000/webhook

  # ─── Logs ──────────────────────────────────────────
  ErrorLog  ${APACHE_LOG_DIR}/postventa-error.log
  CustomLog ${APACHE_LOG_DIR}/postventa-access.log combined
</VirtualHost>

# Redireccion HTTP → HTTPS
<VirtualHost *:80>
  ServerName postventa.baraldoargentina.com.ar
  RewriteEngine On
  RewriteRule ^ https://%{HTTP_HOST}%{REQUEST_URI} [L,R=301]
</VirtualHost>

Habilitacion de modulos necesarios

sudo a2enmod proxy
sudo a2enmod proxy_http
sudo a2enmod ssl
sudo a2enmod rewrite
sudo systemctl restart apache2

# Verificar que no hay errores de configuracion
sudo apache2ctl configtest
Nota de seguridad: Con ProxyPreserveHost On, el header Host original se pasa a Node.js. El bot no depende de este header, pero es buena practica para logging.

10. Certificado SSL con Let's Encrypt

El webhook de Meta requiere HTTPS con certificado válido. Se instaló Certbot con el plugin de Apache para generar y gestionar automáticamente un certificado Let's Encrypt en el dominio postventa.baraldoargentina.com.ar.

✅ Certificado instalado y activo. Válido hasta 2026-09-07. La renovación automática fue configurada por Certbot vía systemd timer.

Requisito previo: DNS

Antes de solicitar el certificado se verificó que el dominio resuelva correctamente a la IP pública del servidor AWS:

dig postventa.baraldoargentina.com.ar +short
# Resultado: 3.221.156.253 (IP pública de la instancia EC2)

curl -s https://checkip.amazonaws.com
# Resultado: 3.221.156.253 ✓ — coincide con el DNS

Instalación de Certbot

sudo apt update
sudo apt install -y certbot python3-certbot-apache

VirtualHost HTTP previo (requerido por Certbot)

Se creó el archivo de configuración del sitio en Apache antes de ejecutar Certbot. El plugin de Apache necesita encontrar un VirtualHost con el ServerName correcto para poder modificarlo y agregar la configuración SSL.

# /etc/apache2/sites-available/postventa.baraldoargentina.com.ar.conf

  ServerName postventa.baraldoargentina.com.ar

  ErrorLog  ${APACHE_LOG_DIR}/postventa-error.log

  # Proxy al bot Node.js
  ProxyPreserveHost On
  ProxyPass        /webhook http://localhost:3000/webhook
  ProxyPassReverse /webhook http://localhost:3000/webhook

sudo a2ensite postventa.baraldoargentina.com.ar.conf
sudo systemctl reload apache2
